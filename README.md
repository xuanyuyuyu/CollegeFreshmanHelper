# 高校新生入学服务与交流平台

## 1. 项目简介

本项目是一款面向高校新生的“入学服务与交流平台”，核心目标是解决新生入学前后的信息获取、经验交流、校园适应和常见问题答疑问题。

平台由两条核心业务主线组成：

- `RAG 智能问答系统`：为新生提供基于高校知识库的智能问答能力。
- `UGC 交流大厅（论坛）`：为新生、老生、管理员提供发帖、回复、互动、经验沉淀能力。

系统具备明显的 `数据飞轮` 特征：

- 老生在论坛中的高质量回答，会在满足条件后被提纯为标准 QA。
- 提纯后的 QA 会写入向量知识库，反哺 RAG 问答能力。
- 问答能力越强，用户提问越集中，论坛内容沉淀也越快。

## 2. 项目目标

### 2.1 用户目标

- 为新生提供入学前后的高频问题解答。
- 为老生提供经验分享、答疑和社区贡献入口。
- 为管理员提供可控、可审计的社区治理能力。

### 2.2 技术目标

- 在 `2C2G` 这类极度受限资源环境中稳定运行。
- 尽量使用轻量但高性能的中间件组合。
- 将论坛与 RAG 知识库打通，形成知识自增长能力。

### 2.3 产品目标

- 第一阶段先完成论坛、认证、管理员治理、基础前端页面。
- 第二阶段接入内容审核、点赞高并发方案、知识飞轮。
- 第三阶段完成完整的 RAG 问答体验与后台知识管理。

## 4. 核心业务闭环

### 4.1 论坛业务闭环

流程：

1. 用户注册并登录平台
2. 用户发帖或回复
3. 内容通过审核后对外展示
4. 其他用户浏览、回复、点赞
5. 管理员可进行隐藏、删除、封禁等治理操作

### 4.2 RAG 业务闭环

流程：

1. 新生向智能问答模块提问
2. 系统从 Qdrant 中检索高质量 QA
3. 将相关 Q+A 作为上下文传给大模型
4. 生成最终回答返回用户

### 4.3 数据飞轮闭环

流程：

1. 老生在论坛回答问题
2. 某条高质量回复达到触发阈值，例如 `like_count == 10`
3. 触发 MQ 消息
4. 消费者将“主贴问题 + 老生回答”提纯为结构化 QA
5. 调用 Embedding 接口生成向量
6. Upsert 写入 Qdrant
7. 回写 MySQL 记录来源、贡献者、奖励积分

## 7. 论坛模块

### 7.1 发帖

功能目标：

- 用户可发布帖子
- 支持纯文本帖
- 支持图文帖
- 支持标签
- 自动记录作者、发布时间、状态

### 7.2 帖子列表

功能目标：

- 分页展示帖子
- 展示标题、作者、标签、回复数、点赞数、发布时间
- 仅展示可见且已发布的帖子

### 7.3 帖子详情

功能目标：

- 展示帖子完整内容
- 展示图片列表
- 展示回复线程
- 展示帖子统计信息

### 7.4 回复系统

功能目标：

- 支持一级回复
- 支持对任意一条回复进行再次回复
- 数据结构保持双层展示，不做无限缩进

当前设计语义：

- 一级回复：`parent_id = 0`
- 楼中楼回复：`parent_id = 所属一级回复ID`
- 实际被回复目标：`reply_to_reply_id`
- 实际被回复用户：`reply_to_user_id`

这意味着：

- 用户 3 可以回复“用户 1 回复用户 2”的那条回复
- 但前端展示依然保持一级楼层 + 子回复列表的双层结构

### 7.5 我的帖子 / 我的回复

计划功能：

- 用户可查看自己发布的帖子
- 用户可查看自己发表的回复
- 用户可查看自己收到的互动情况

## 8. 点赞与互动模块

### 8.1 帖子点赞

目标规则：

- 用户可以点赞帖子
- 用户不能重复点赞
- 用户可以取消点赞

### 8.2 回复点赞

目标规则：

- 用户可以点赞回复
- 用户不能重复点赞
- 用户可以取消点赞

### 8.3 高并发实现约束

必须遵守：

- 点赞不能直接 `UPDATE MySQL`
- 点赞先写 Redis
- 使用 `Redis Set / Bitmap` 记录点赞关系
- 使用定时任务批量回刷 MySQL

### 8.4 收藏与热门内容

计划功能：

- 帖子收藏
- 热门帖子缓存
- 热门排序

## 14. 内容安全模块

> 该模块属于核心红线模块，但可以在阶段上后置开发。

### 14.1 纯文本审核

目标规则：

- 发帖和回复前，在 Service 层同步调用阿里云 Green
- 审核不通过则直接阻断落库

### 14.2 图文帖审核

目标规则：

- 图片必须先上传 OSS
- 图文帖落库时状态设为 `status=0`
- MQ 异步触发图片检测
- 仅审核通过后才能切换为 `status=1`

### 14.3 管理员联动

目标能力：

- 对违规内容进行下线
- 保留违规原因
- 形成审计链路

## 19. 极限资源约束

部署约束：

- 服务器规格：`2C2G`
- JVM：`-Xms256m -Xmx512m`
- MySQL：`innodb_buffer_pool_size = 128M`
- Qdrant：Docker `mem_limit: 300M`，启用 On-Disk 模式

设计原则：

- 单体优先
- 强异步
- 少依赖
- 尽量避免重型组件
- 点赞、审核、提纯、向量写入全部解耦
